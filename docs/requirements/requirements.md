# Разработка модуля автоматизации складского учета
## Требования на разработку системы
Я, как управляющий склада, хочу автоматизировать складской учет для быстрой обработки товаров и оптимизации складских запасов.

Роль: управляющий склада

Цель: автоматизация складского учета

История: Я, как управляющий склада, хочу иметь возможность вести учет товаров на складе, автоматически обновлять остатки при поступлении и отправке товара, а также генерировать отчеты по движению товаров за определенный период и прогнозировать остатки товаров для своевременного пополнения.

### Критерии приемки: 
1. Система строит дашборд по выбранной зоне хранения, где отображает текущее количество товаров и прогноз по остаткам через неделю.
2. Система автоматически обновляет остатки по зонам склада (поступление, хранение, отправка) при поступлении и отправке товара, чтобы данные на дашборде были актуальны.
4. Система генерирует Excel-отчёты по периоду с суммарными остатками, отправками за прошлую неделю и прогнозом на неделю.
5. Пользователь может сам запрашивать отчет с последующей отправкой на почту
6. Система по расписанию делает рассылку отчетов подписанным пользователям.
7. Система перемещает товар между зонами согласно заказу.
8. Система прогнозирует остатки товаров через неделю.

## Описание Use Cases

### Use Case 1: Просмотр дашборда зоны

Название: Просмотр дашборда

Участники: управляющий склада

Предусловия: управляющий авторизован, открыта панель администрирования

Основной сценарий:
- Управляющий выбирает зону склада в панели администрирования.
- Система отображает страницу выбранной зоны.
- Управляющий открывает дашборд зоны.
- Система формирует данные для дашборда.
- Система отображает управляющему: текущее количество товаров в выбранной зоне; прогноз суммарного остатка по складу через неделю.
- Управляющий просматривает информацию на дашборде.

Альтернативные сценарии:
- Нет данных: Система отображает сообщение об ошибке «Не удалось получить данные» и предлагает повторить попытку.
- Нет прогноза: Система отображает текущие данные зоны. В поле прогноза отображается сообщение: «Прогноз временно недоступен.»

### Use Case 2: Обработка заказа из внешнего сервиса

Название: Обработка заказа

Участники: внешний сервис учета заказов

Предусловия: внешний сервис сформировал заказ, интеграция между системами настроена.

Основной сценарий:
- Внешний сервис отправляет информацию о заказе в систему склада.
- Система проверяет возможность создания заказа и сразу возвращает ответ, что заказ принят.
- После создания заказа система проверяет наличие всех товаров в нужном количестве.
- Если товары есть, система резервирует их за заказом.
- Система склада возвращает внешнему сервису подтверждение о том, что товары зарезервированы.
- Система создаёт движение товаров между зонами склада.
- Система обновляет текущие остатки товаров.

Альтернативные сценарии: 
- Недостаток товара: Система уведомляет внешний сервис о нехватке товара и заказ отклоняется.
- Некорректные данные заказа: Система отправляет сообщение об ошибке внешнему сервису, заказ не принимается.
- Технический сбой: система возвращает временную ошибку, внешний сервис может повторить запрос позже.

### Use Case 3: Запрос отчета пользователем
Название: Формирование отчета по запросу

Участники: Управляющий склада

Предусловия: управляющий авторизован, панель администрирования открыта.

Основной сценарий:
- управляющий нажимает кнопку «Создать отчет».
- Система предлагает выбрать период (дата начала, дата конца).
- Управляющий вводит период и подтверждает формирование отчёта.
- Система проверяет корректность периода.
- Система регистрирует запрос.
- Система выводит пользователю сообщение о том, что отчет формируется и придет на почту.
- Система асинхронно формирует отчёт и отправляет его на почту управляющему.

Альтернативные сценарии:
- Неправильный период: Система сообщает об ошибке ввода периода и предлагает ввести корректные даты.
- Нет данных за выбранный период: система формирует пустой отчёт с соответствующим уведомлением.
- Технический сбой: система показывает ошибку и предлагает повторить.

### Use Case 4: Генерация отчета по расписанию 
Название: Автоматическая генерация отчета по расписанию

Участники: Планировщик (Scheduler) — внутренний компонент системы, срабатывающий по заданному расписанию.

Предусловия: Настроено расписание (например, каждый понедельник в 9:00).

Основной сценарий:
- В заданное время планировщик инициирует событие генерации отчета.
- Система определяет период отчета (например, последние 7 дней).
- Система собирает данные о движении товаров и текущих остатках за указанный период.
- Система формирует Excel-файл с отчётом.
- Система сохраняет отчёт в хранилище и передаёт его сервису уведомлений для отправки подписанным пользователям.

Альтернативные сценарии:
- Данные за период отсутствуют: Система создаёт пустой отчёт с пометкой «Нет данных за выбранный период».
- Ошибка формирования файла: Система логирует ошибку и повторяет попытку через заданный интервал (или отправляет уведомление администратору).

### Use Case 5: Прогнозирование остатков
Название: Прогнозирование остатков

Участники: сервис ML-модели.

Предусловия: Настроено расписание (раз в сутки обновляется прогноз).

Основной сценарий:
- Планировщик инициирует задачу расчёта прогноза.
- Сервис склада возвращает в ML-сервис агрегированные исторические данные по товарам за период: остаток товара, количество поступившего товара и количество отправленного товара по дням.
- ML-сервис применяет модель прогнозирования временных рядов.
- Рассчитывается прогноз остатков через 7 дней от текущей даты.
- Результаты сохраняются в Redis (для дашборда) и сохраняются в таблицу Forecasts (для истории)

Альтернативные сценарии:
- Недостаточно данных: Система логирует ситуацию, прогноз не обновляется.
- Ошибка модели: Система логирует ошибку, прогноз остаётся прежним.
- Ошибка запроса данных: повтор запроса.

### Use Case 6: Переобучение ML-модели
Название: Переобучение модели прогнозирования

Актор: Планировщик (Scheduler)

Предусловия: накоплены исторические данные о движении и остатках товаров, ML-сервис доступен, настроено расписание (раз в неделю обучается модель).

Основной сценарий:
- В заданное время (например, раз в неделю) планировщик инициирует задачу переобучения модели.
- ML-сервис запрашивает исторические данные у сервиса склада за длительный период.
- Сервис склада извлекает данные из БД и передаёт их ML-сервису.
- ML-сервис подготавливает данные: агрегирует по временным интервалам (например, по дням), очищает и нормализует данные
- ML-сервис обучает новую версию модели прогнозирования.
- Обученная модель сохраняется и помечается как актуальная.
- Система логирует успешное завершение обучения.

Альтернативные сценарии:
- Недостаточно данных: обучение не выполняется, используется текущая модель
- Ошибка обучения: система логирует ошибку, предыдущая версия модели остаётся активной
- Ошибка получения данных: повтор попытки через заданный интервал.


## Нефункциональные требования

- Время ответа API не более 2 секунд.
- Система должна обеспечивать доступность 99%.
- Система должна поддерживать горизонтальное масштабирование.
- Система должна обеспечивать обработку не менее 1000 заказов в сутки.
- Система должна сохранять работоспособность при временной недоступности сервиса прогнозирования.
- Система должна поддерживать повторную доставку событий Kafka при временных сбоях потребителей.
